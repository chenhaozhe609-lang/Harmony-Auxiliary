# Task 2: Rhythm-Aware Melody Editing and Harmony Alignment

## 1. 背景

这次调整来自试用反馈。当前版本已经可以导入或手动输入旋律，生成和声候选，播放并导出 MIDI；但试用者认为主要问题集中在节奏逻辑、旋律输入方式、和声显示方式，以及古典和声规则的准确性。

核心反馈可以归纳为：

- 旋律音符需要进入更细的节奏格，而不是只靠顺序追加。
- 用户需要通过点击或拖动来输入、移动、调整旋律音符。
- 和弦不应只显示为一个块，需要展开为多个同时发声的音轨，并在下方标注和弦名字。
- 和声节奏需要可选，例如只在重拍换和弦、在重拍和次重拍换和弦等。
- 古典风格需要更遵守古典和声规律。
- 旋律和和声在图示、播放、生成逻辑上需要准确对齐。
- 生成内容应该支持更长的旋律，而不是像只适合短片段演示。

本轮任务的目标不是把产品变成完整 DAW 或完整五线谱编辑器，而是把现有 MVP 从“能生成和弦”推进到“能按节奏组织旋律与和声”。

## 2. 产品目标

Task 2 的目标是建立一个节奏感更清晰、显示更可信、输入更自然的工作区：

1. 用户能在可见的节奏网格里输入旋律。
2. 音符的起点和时值能准确显示。
3. 和声生成能根据用户选择的和声节奏进行分段。
4. 和弦音能展开显示为多个音轨。
5. 和弦名称、罗马数字、功能标签与实际时间位置对齐。
6. Stable Classical 模式要更像古典功能和声，而不是只按局部音符匹配。

## 3. 非目标

本轮不做以下内容：

- 完整五线谱编辑。
- MusicXML 导入或导出。
- 完整四部和声批改。
- 复杂 DAW 级 piano roll。
- AI 聊天式作曲。
- 云端项目、账户、协作功能。

如果需要五线谱，本轮最多做只读或参考显示，不作为主要编辑面。

## 4. 任务拆分

### T2.1 节奏网格与动态小节尺

目标：让时间线真正表达拍点、小节、细分单位和长旋律。

任务：

- 将时间线从固定 4 小节显示改为根据旋律和和声总长度动态计算。
- 根据拍号生成小节尺，例如 4/4 下显示 bar 1, bar 2, bar 3。
- 增加节奏细分网格：
  - 每拍。
  - 半拍。
  - 八分音符格。
- 音符块宽度由 `durationBeats` 决定。
- 音符块位置由 `startBeat` 决定。
- 支持较长旋律的横向滚动或缩放，不让页面整体溢出。
- 播放头需要和网格对齐。

验收标准：

- 8 小节以上旋律能完整显示和滚动查看。
- 音符起点、时值、播放头位置在视觉上和实际播放一致。
- 导入 MIDI 后，音符不会被压缩成只能看见前 4 小节。

### T2.2 点击与拖动输入旋律

目标：替代单纯“追加音符”的手动输入方式，让用户能在指定节奏位置放置音符。

任务：

- 保留原有 note-name 快速输入作为简化入口。
- 增加 piano-roll 式编辑区域。
- 用户点击某个 pitch row 和 beat cell 时创建音符。
- 用户可以拖动音符改变 `startBeat`。
- 用户可以拖动音符右边缘改变 `durationBeats`。
- 拖动结果需要吸附到当前节奏细分格。
- 增加选中音符状态。
- 支持删除选中音符。
- 支持 undo / clear melody 继续工作。

验收标准：

- 用户不依赖 MIDI 文件，也能创建一段有节奏逻辑的旋律。
- 点击创建、拖动移动、拖动改时值后的音符能正确播放。
- 生成和声时使用的是编辑后的真实 `startBeat` 和 `durationBeats`。

### T2.3 和声节奏模式

目标：把原来的 `bar / half-bar` 和声密度升级为明确的和声节奏模式。

建议模式：

- `bar`: 每小节一个和弦。
- `strong-beats`: 重拍和次强拍换和弦，例如 4/4 中 beat 1 和 beat 3。
- `every-beat`: 每拍都允许换和弦。
- `cadence-aware`: 平时每小节或半小节，句尾自动加密以形成终止式。
- `sparse`: 长和弦，适合慢速或旋律音较多的片段。

任务：

- 新增 `HarmonyRhythmPattern` 类型。
- 将 `ProjectSettings.harmonyDensity` 迁移或扩展为 `harmonyRhythmPattern`。
- 改造 melody segmentation，使它根据节奏模式生成 segment。
- 让 UI 中的“和声节奏”显示为可理解的选项，而不是抽象 density。
- 每个 segment 保留准确的 `startBeat` 和 `durationBeats`。
- 生成、播放、导出、解释都使用同一个 segment 结果。

验收标准：

- 用户选择不同和声节奏后，生成的和弦数量和位置明显不同。
- 在 4/4 中选择“重拍和次强拍”时，和弦应主要落在 beat 1 和 beat 3。
- 播放、图示、MIDI 导出中的和弦位置一致。

### T2.4 和声音轨展开显示

目标：让和弦不再只是一个文字块，而是能看见它由多个音组成。

任务：

- 将当前 chord lane 拆为：
  - Melody lane。
  - Harmony voice lanes。
  - Chord label lane。
- 和声 voice lanes 可以先使用简单固定声部：
  - Bass。
  - Lower voice。
  - Middle voice。
  - Upper voice。
- 每个和弦音显示为横向音符块，位置和长度与 placed chord 对齐。
- Chord label lane 在和弦音轨下方显示和弦名。
- 和弦名旁继续显示 Roman numeral 或功能标签。
- 点击和弦名或和弦音块时，选中同一个 placed chord，并更新 inspector。

验收标准：

- 用户能看到一个和弦对应多个同时发声的音。
- 和弦名显示在对应和声片段下方。
- 选中和弦后，展开音轨、chord label、inspector 三者状态一致。

### T2.5 旋律与和声对齐修正

目标：解决“旋律和和声没有准确按照图示对上”的问题。

任务：

- 定义统一的 beat-to-position 映射函数。
- 时间线显示、播放头、音符位置、和弦位置都使用同一套映射。
- 消除 UI 中固定 4 小节、固定 16 格等硬编码假设。
- 增加边界测试：
  - 音符从 beat 0 开始。
  - 音符从弱拍开始。
  - 跨小节长音。
  - 半拍和八分音符。
  - 8 小节以上旋律。
- 检查 MIDI import 的 beat normalization 是否和网格一致。
- 检查 MIDI export 的 ticks 是否和 UI beat 一致。

验收标准：

- 同一个音符在 UI、播放、导出 MIDI 中位置一致。
- 半拍音符和跨小节音符不会显示错位。
- 视觉和听觉上的重拍位置一致。

### T2.6 Stable Classical 古典规则增强

目标：让古典候选不只是逐段挑最高分和弦，而是具有基本功能和声句法。

任务：

- 为 Stable Classical 增加功能走向评分：
  - T -> PD。
  - PD -> D。
  - D -> T。
  - T -> T substitute。
  - PD -> PD substitute。
- 在句尾或乐段末尾优先考虑终止式：
  - V -> I。
  - IV -> V -> I。
  - ii -> V -> I。
- 强拍上的长旋律音优先作为和弦音处理。
- 弱拍短音允许作为经过音或邻音。
- 对连续不合逻辑的功能跳转加惩罚。
- 对长时值非和弦音加更明确的 warning。
- 在 explanation 中说明：
  - 当前和弦支持了哪个旋律音。
  - 当前和弦承担什么功能。
  - 它与前后和弦的关系是否形成常见功能进行。

验收标准：

- Stable Classical 结果比 Pop / Color 更稳定、更功能化。
- 末尾能更容易生成清楚的 V-I 或 ii-V-I。
- inspector 能解释和弦为什么适合当前旋律和当前功能位置。

### T2.7 长旋律支持

目标：让产品可以处理更长一点的旋律，而不是只像四小节 demo。

任务：

- 将 demo melody 扩展或新增一个 longer demo。
- UI 支持至少 8-16 小节查看。
- Candidate strip 中长和弦进行需要截断显示，但不丢失完整信息。
- Copy progression 仍复制完整和弦进行。
- Playback end beat 使用旋律和和声最大结束点。
- 生成过程需要避免因为 segment 过多导致 UI 卡顿。

验收标准：

- 12 小节左右旋律可以生成、播放、查看、复制和导出。
- 长进行在候选卡片中不会撑坏布局。
- inspector 能查看任意位置的和弦。

### T2.8 测试与验证

目标：保证节奏、显示、播放、导出不会各算各的。

任务：

- 添加 segment generation 单元测试。
- 添加 harmony rhythm pattern 测试。
- 添加 beat-to-grid mapping 测试。
- 添加 chord voice lane conversion 测试。
- 添加 MIDI export timing 测试。
- 添加至少一个长旋律 fixture。
- 增加浏览器验证脚本：
  - 打开 workspace。
  - 加载 demo 或 longer demo。
  - 切换和声节奏。
  - 生成候选。
  - 检查 timeline 没有横向页面溢出。
  - 检查 melody lane 和 chord label lane 有可见内容。

验收标准：

- `npm test` 通过。
- `npm run build` 通过。
- 浏览器验证显示时间线、播放头、和弦标签、和声音轨都正常可见。

## 5. 推荐实施顺序

建议按以下顺序开发：

1. T2.1 节奏网格与动态小节尺。
2. T2.3 和声节奏模式。
3. T2.5 旋律与和声对齐修正。
4. T2.4 和声音轨展开显示。
5. T2.2 点击与拖动输入旋律。
6. T2.6 Stable Classical 古典规则增强。
7. T2.7 长旋律支持。
8. T2.8 测试与验证。

原因：

- 节奏网格和和声节奏是地基。
- 对齐修正应在 UI 展开前完成。
- 和声音轨展开依赖统一的时间映射。
- 拖动编辑依赖稳定的网格和吸附规则。
- 古典规则增强依赖更准确的 segment。

## 6. 风险与注意事项

- 不要在本轮尝试完整五线谱编辑，否则范围会失控。
- 不要让 UI、播放、导出分别计算时间位置，必须共享 beat-based 数据。
- 不要只增加视觉网格而不改生成逻辑，否则“看起来有节奏，但生成仍没逻辑”。
- 古典模式需要保守，不应为了显得丰富而过度使用色彩和弦。
- 对齐问题要用 fixture 和浏览器验证解决，不能只靠肉眼临时调 CSS。

## 7. 完成定义

Task 2 完成时，用户应该能：

- 在节奏网格里看到清楚的小节和拍点。
- 输入或导入一段较长旋律。
- 选择不同和声节奏。
- 生成与节奏位置对齐的和声。
- 看到和弦展开成多个音轨。
- 在和弦下方看到准确的和弦名和功能信息。
- 播放时听到旋律和和声与图示一致。
- 导出 MIDI 后，在外部工具中看到相同的时间结构。
